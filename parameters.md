# Field Reference — AWS Network Firewall & VPC Flow Logs

All fields observable in this POC, grouped by log source. Fields marked ⚙️ are set by your
Suricata rules; fields marked 🔧 are generated by the NFW/Suricata engine automatically.

---

## 1. NFW Alert Log (`/trafinspector/nfw/alert`)

Emitted when a Suricata rule with action `alert` or `drop` matches. Source: **Suricata IDS engine**
running inside AWS Network Firewall.

### Envelope (AWS-added)

| Field | Source | Description |
|-------|--------|-------------|
| `firewall_name` | 🔧 NFW | Name of the firewall resource (`trafinspector-fw`) |
| `availability_zone` | 🔧 NFW | AZ where the NFW endpoint processed this packet |
| `event_timestamp` | 🔧 NFW | Unix epoch seconds when the event was logged |

### Event — Connection Identity

| Field | Source | Description |
|-------|--------|-------------|
| `event.src_ip` | 🔧 Suricata | Source IP (original, pre-NAT — from GENEVE inner header) |
| `event.dst_ip` / `event.dest_ip` | 🔧 Suricata | Destination IP |
| `event.src_port` | 🔧 Suricata | Source port (ephemeral) |
| `event.dest_port` | 🔧 Suricata | Destination port |
| `event.proto` | 🔧 Suricata | IP protocol (`TCP`, `UDP`, `ICMP`) |
| `event.flow_id` | 🔧 Suricata | **Unique flow identifier** — see §4 below |
| `event.pkt_src` | 🔧 NFW | Packet source (`geneve encapsulation` = arrived via GWLB/TGW) |
| `event.direction` | 🔧 Suricata | `to_server` or `to_client` |
| `event.event_type` | 🔧 Suricata | Always `alert` for this log type |

### Event — Alert (rule match)

| Field | Source | Description |
|-------|--------|-------------|
| `event.alert.signature_id` | ⚙️ Your rule `sid:` | Integer you assign per rule to uniquely identify it. Convention: 1–999999 local, 2000000+ community (ET). |
| `event.alert.signature` | ⚙️ Your rule `msg:` | Human-readable label ("Zoom", "MS Teams", "Block QUIC") |
| `event.alert.rev` | ⚙️ Your rule `rev:` | Rule revision number (increment when you edit a rule) |
| `event.alert.severity` | ⚙️ Your rule `priority:` / default | 1=highest, 4=lowest. Defaults: `drop`→2, `alert`→3 if no `priority:` keyword set |
| `event.alert.action` | 🔧 Suricata | Action taken: `allowed` (alert only) or `blocked` (drop rule) |
| `event.alert.category` | ⚙️ Your rule `classtype:` | Rule category string (empty if not set). E.g. `trojan-activity`, `policy-violation` |

### Event — Verdict

| Field | Source | Description |
|-------|--------|-------------|
| `event.verdict.action` | 🔧 Suricata | Final verdict: `alert` (logged, passed) or `drop` (blocked) |

### Event — Application Protocol

| Field | Source | Description |
|-------|--------|-------------|
| `event.app_proto` | 🔧 Suricata | Detected L7 protocol: `tls`, `http`, `http2`, `dns`, `smtp`, `ftp`, `ssh`, `smb`, etc. |
| `event.tx_id` | 🔧 Suricata | Transaction index within the flow (0-based; increments per HTTP request/response pair) |

### Event — TLS metadata (present when `app_proto` = `tls`)

| Field | Source | Description |
|-------|--------|-------------|
| `event.tls.sni` | 🔧 Suricata | Server Name Indication from ClientHello — the hostname the client requested |
| `event.tls.version` | 🔧 Suricata | TLS version (`TLS 1.2`, `TLS 1.3`, `UNDETERMINED` if handshake incomplete) |
| `event.tls.subject` | 🔧 Suricata | Server certificate subject DN |
| `event.tls.issuer` | 🔧 Suricata | Server certificate issuer DN |
| `event.tls.fingerprint` | 🔧 Suricata | SHA-1 fingerprint of the server certificate |
| `event.tls.serial` | 🔧 Suricata | Certificate serial number |
| `event.tls.notbefore` | 🔧 Suricata | Certificate validity start |
| `event.tls.notafter` | 🔧 Suricata | Certificate validity end |

### Event — HTTP metadata (present when TLS decryption is active → `app_proto` = `http2`/`http`)

| Field | Source | Description |
|-------|--------|-------------|
| `event.http.hostname` | 🔧 Suricata | HTTP Host header / `:authority` pseudo-header (HTTP/2) |
| `event.http.url` | 🔧 Suricata | Request path + query string |
| `event.http.http_method` | 🔧 Suricata | `GET`, `POST`, `PUT`, etc. |
| `event.http.http_user_agent` | 🔧 Suricata | User-Agent header — strong app identifier |
| `event.http.http_content_type` | 🔧 Suricata | Response Content-Type |
| `event.http.http_refer` | 🔧 Suricata | Referer header |
| `event.http.protocol` | 🔧 Suricata | HTTP version (`HTTP/1.1`, `HTTP/2`) |
| `event.http.status` | 🔧 Suricata | Response status code (200, 301, 403, etc.) |
| `event.http.length` | 🔧 Suricata | Response body length in bytes |

### Event — AWS category (NFW managed rule groups only)

| Field | Source | Description |
|-------|--------|-------------|
| `event.aws_category` | 🔧 NFW | Domain category from AWS managed domain lists (e.g. `malware`, `botnet`). Empty for custom rules. |

| `firewall_name` | 🔧 NFW | Firewall name |
| `availability_zone` | 🔧 NFW | Processing AZ |
| `event_timestamp` | 🔧 NFW | Unix epoch |
| `event.event_type` | 🔧 Suricata | Always `netflow` |
| `event.src_ip` | 🔧 Suricata | Source IP |
| `event.dest_ip` | 🔧 Suricata | Destination IP |
| `event.src_port` | 🔧 Suricata | Source port |
| `event.dest_port` | 🔧 Suricata | Destination port |
| `event.proto` | 🔧 Suricata | Protocol |
| `event.flow_id` | 🔧 Suricata | Flow identifier — correlates with alert/tls logs |

| `event.tls_inspected` | 🔧 NFW | Whether TLS decryption was applied |
| `event.tcp.tcp_flags` | 🔧 Suricata | Hex bitmask of all TCP flags seen in the flow |

| `event.tcp.ack` | 🔧 Suricata | ACK flag seen |
| `event.netflow.pkts` | 🔧 Suricata | Total packets in this direction |
| `event.netflow.bytes` | 🔧 Suricata | Total bytes in this direction |
| `event.netflow.start` | 🔧 Suricata | Flow start timestamp (ISO 8601) |

**Note:** NFW emits **two netflow records per connection** — one for each direction (client→server, server→client), sharing the same `flow_id`.

## 4. How `flow_id` is generated

`flow_id` is generated by **Suricata's flow engine** — not by AWS, not by your rules.

Suricata assigns a `flow_id` when it first sees a new 5-tuple (src_ip, dst_ip, src_port, dst_port, proto).
The value is a **64-bit integer derived from an internal hash + monotonic counter** — it's unique within
a single Suricata instance's lifetime (i.e., per NFW endpoint, per reboot).

**Correlation:** The same `flow_id` appears across all three NFW log types for the same TCP/UDP connection:
- Alert log → which rules matched
- TLS log → certificate/handshake details
- Flow log → byte/packet counters and duration

This is the **join key** across NFW log types. Example: `flow_id: 1430312426565211` in an alert log
will have matching entries in the TLS and flow logs with the same value.

**Scope:** `flow_id` is unique per AZ endpoint. Two different AZ endpoints could theoretically produce
the same `flow_id` for different flows, so always include `availability_zone` in joins.

## 5. VPC Flow Logs (`/trafinspector/flowlogs`)

Custom format configured in `flowlogs.tf`. Source: **AWS VPC Flow Logs service** (kernel-level capture).

| Position | Field | Source | Description |
|----------|-------|--------|-------------|
| 1 | `srcaddr` | 🔧 VPC | Source IP as seen by the ENI (may be NAT'd / TGW-translated) |
| 2 | `dstaddr` | 🔧 VPC | Destination IP as seen by the ENI |
| 3 | `srcport` | 🔧 VPC | Source port |
| 4 | `dstport` | 🔧 VPC | Destination port |
| 5 | `protocol` | 🔧 VPC | IANA protocol number (6=TCP, 17=UDP, 1=ICMP) |
| 6 | `pkt-srcaddr` | 🔧 VPC | **Original** source IP from the packet header (survives NAT/TGW translation) |
| 7 | `pkt-dstaddr` | 🔧 VPC | **Original** destination IP from the packet header |
| 8 | `bytes` | 🔧 VPC | Total bytes transferred in the aggregation interval |
| 9 | `packets` | 🔧 VPC | Total packets in the aggregation interval |
| 10 | `action` | 🔧 VPC | `ACCEPT` or `REJECT` (security group / NACL verdict) |
| 11 | `flow-direction` | 🔧 VPC | `ingress` or `egress` relative to the ENI |
| 12 | `start` | 🔧 VPC | Unix epoch of aggregation window start |
| 13 | `end` | 🔧 VPC | Unix epoch of aggregation window end |

## 6. Key differences: NFW vs VPC Flow Logs

| Capability | NFW (Suricata) | VPC Flow Logs |
|-----------|----------------|---------------|
| Layer | L3–L7 (deep packet inspection) | L3–L4 only |
| TLS decryption | ✅ (MITM via CA) | ❌ |
| SNI visibility | ✅ (from ClientHello) | ❌ |
| HTTP headers | ✅ (when decrypted) | ❌ |
| Application detection | ✅ (protocol + rules) | ❌ |
| Per-flow unique ID | ✅ (`flow_id`) | ❌ (no equivalent) |
| Original IPs behind NAT | ✅ (GENEVE inner) | ✅ (`pkt-srcaddr`/`pkt-dstaddr`) |
| Byte/packet counts | ✅ (per direction) | ✅ (aggregated per interval) |
| Security group verdict | ❌ | ✅ (`action` = ACCEPT/REJECT) |
| JA3/JA4 fingerprints | ❌ (not enabled in NFW by default) | ❌ |
| Aggregation | Per-flow (on close/timeout) | Per-interval (60s in this POC) |

## 7. Correlating NFW ↔ VPC Flow Logs

There is **no shared flow ID** between NFW and VPC Flow Logs. Join on the **5-tuple**:

```
NFW:       (event.src_ip, event.dest_ip, event.src_port, event.dest_port, event.proto)
Flow Logs: (pkt-srcaddr, pkt-dstaddr, srcport, dstport, protocol)
```

Use `pkt-srcaddr`/`pkt-dstaddr` (not `srcaddr`/`dstaddr`) because the latter reflect the
ENI-local view after TGW/NAT translation, while NFW sees the original packet IPs via GENEVE.

Time overlap: match `event.netflow.start`/`end` against flow log `start`/`end` windows.
